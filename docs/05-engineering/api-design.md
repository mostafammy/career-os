**Status:** Canonical
**Version:** 1.0

## 1. Purpose

This document defines the API surface for CareerOS. It specifies endpoints, request/response schemas, authentication, error handling, and versioning strategy.

The API follows domain-oriented design (ref: P-009 — Design Around Intent). Endpoints are named after business actions, not CRUD operations.

---

## 2. API Philosophy

### 2.1 Domain-Oriented Design

Instead of generic CRUD:

```
POST   /api/opportunities          (Create)
GET    /api/opportunities/:id      (Read)
PATCH  /api/opportunities/:id      (Update)
DELETE /api/opportunities/:id      (Delete)
```

CareerOS exposes intent-based actions:

```
POST   /api/opportunities/capture          (Capture Opportunity)
POST   /api/opportunities/:id/evaluate     (Start Evaluation)
POST   /api/opportunities/:id/action       (Action Opportunity)
POST   /api/opportunities/:id/archive      (Archive Opportunity)
POST   /api/opportunities/:id/restore      (Restore Opportunity)
```

**Rationale:** The API language matches the user's mental model (P-009), the domain ubiquitous language (glossary.md), and the state machines (state-machines.md).

### 2.2 API Principles

| ID | Principle | Rationale |
|---|---|---|
| API-001 | **REST-first** | Familiar, tooling support, cacheable |
| API-002 | **Versioned** | Breaking changes are managed, not feared |
| API-003 | **Documented** | OpenAPI/Swagger for every endpoint |
| API-004 | **Idempotent** | Safe retries for all mutation operations |
| API-005 | **Secure** | Every endpoint requires authentication |
| API-006 | **Consistent** | Uniform response format, error structure, pagination |
| API-007 | **Domain-oriented** | Named after business actions, not database operations |

## 3. Base Configuration

### 3.1 Base URL

```
Production:  https://api.career-os.mostafayaser.earth/v1
Development: http://localhost:3000/api/v1
```

### 3.2 Authentication

All API endpoints require authentication via session cookie (NextAuth.js).

```
Cookie: next-auth.session-token=<token>
```

Unauthenticated requests receive `401 Unauthorized`.

### 3.3 Content Type

```
Content-Type: application/json
Accept: application/json
```

### 3.4 Request ID

Every request includes a unique ID for tracing:

```
X-Request-ID: <uuid-v7>
```

Generated by the server if not provided by the client.

| HTTP Status | Code | Meaning |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Request body validation failed |
| 401 | `UNAUTHORIZED` | Authentication required |
| 403 | `FORBIDDEN` | Insufficient permissions |

| 429 | `RATE_LIMITED` | Too many requests |

## 5. Endpoint Catalog

### 5.1 Opportunity Endpoints

| Method | Endpoint | Action | Description |
|---|---|---|---|
| `POST` | `/opportunities/capture` | Capture | Capture a new opportunity from URL, text, or manual entry |
| `GET` | `/opportunities` | List | List opportunities with filters |
| `GET` | `/opportunities/:id` | Get | Get a single opportunity with full context |
| `POST` | `/opportunities/:id/evaluate` | Evaluate | Move opportunity to EVALUATING state |
| `POST` | `/opportunities/:id/action` | Action | Move opportunity to ACTIONED, create Application |
| `POST` | `/opportunities/:id/archive` | Archive | Move opportunity to ARCHIVED state |
| `POST` | `/opportunities/:id/restore` | Restore | Restore archived opportunity to CAPTURED |
| `PATCH` | `/opportunities/:id` | Update | Update opportunity metadata (title, deadline, etc.) |
| `DELETE` | `/opportunities/:id` | Soft Delete | Soft-delete an opportunity |

#### `POST /opportunities/capture`

Captures a new opportunity. Minimal data required (P-002: Capture First, Organize Later).

**Request:**

```json
{
  "source": "manual" | "extension" | "email" | "import",
  "url": "https://...",           // Optional — original URL
  "raw_text": "...",              // Optional — raw text from page
  "title": "DAAD Scholarship",    // Optional — can be extracted by AI
  "organization_name": "...",     // Optional — can be extracted by AI
  "deadline": "2028-06-15",       // Optional — ISO 8601 date
  "opportunity_type": "scholarship",
  "notes": "..."                  // Optional
}
```

**Response (201):**

```json
{
  "data": {
    "id": "01JAF4YZPQ3T0Q2M7G5BXQ1D8R",
    "title": "DAAD Scholarship",
    "status": "CAPTURED",
    "organization": null,
    "deadline": "2028-06-15",
    "source": "manual",
    "captured_at": "2028-03-12T14:30:00Z"
  }
}
```

**Side Effects:**
- Activity record created
- AI extraction triggered asynchronously (if URL or raw_text provided)

#### `POST /opportunities/:id/action`

Moves opportunity to ACTIONED. Creates Application in DRAFTING state (XAG-001).

**Request:**

```json
{
  "goal_ids": ["goal-1", "goal-2"]  // Optional — for strategic alignment scoring
}
```

**Response (200):**

```json
{
  "data": {
    "opportunity": {
      "id": "01JAF4YZPQ3T0Q2M7G5BXQ1D8R",
      "status": "ACTIONED",
      "actioned_at": "2028-03-15T10:00:00Z"
    },
    "application": {
      "id": "01JAF4YZPQ3T0Q2M7G5BXQ1D8S",
      "status": "DRAFTING",
      "created_at": "2028-03-15T10:00:00Z"
    }
  }
}
```

**Side Effects:**
- Application entity created in DRAFTING state
- Tasks generated from opportunity requirements
- Activity records created for both Opportunity and Application

**Validation:**
- Opportunity must be in EVALUATING state
- If goal_ids provided, goals must exist and be ACTIVE

#### `POST /applications/:id/submit`

Marks application as submitted. Locks submission timestamp (APP-002).

**Request:**

```json
{
  "submitted_at": "2028-03-20T15:00:00Z",  // Optional — defaults to now
  "notes": "Submitted via university portal"
}
```

**Response (200):**

```json
{
  "data": {
    "id": "01JAF4YZPQ3T0Q2M7G5BXQ1D8S",
    "status": "SUBMITTED",
    "submitted_at": "2028-03-20T15:00:00Z"
  }
}
```

**Side Effects:**
- Submission timestamp locked
- Follow-up reminder scheduled (14 days)
- Activity record created

**Validation:**
- Application must be in DRAFTING state
- At least one document must be attached (APP-001)

## 7. Idempotency

All mutation endpoints support idempotency via the `Idempotency-Key` header:

```
Idempotency-Key: <uuid-v7>
```

- If a request with the same key is replayed, the original response is returned
- Idempotency keys expire after 24 hours
- Only applicable to `POST`, `PATCH`, `DELETE` methods

## 9. Versioning Strategy

### 9.1 URL Versioning

```
/v1/opportunities
/v2/opportunities  (future, when breaking changes needed)
```

### 9.2 Versioning Rules

- **Minor changes** (new fields, new endpoints) — Added to current version without version bump
- **Breaking changes** (removed fields, changed semantics) — New version required
- **Deprecated endpoints** — Maintain for 6 months after new version release
- **Sunset header** — `Sunset: Sat, 01 Sep 2028 00:00:00 GMT` for deprecated endpoints
